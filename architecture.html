1. High-Level Architecture Overview

1.1 Current Architecture (v1.3)

The REDCap Delphi Module is a presentation-only feedback layer. It reads manually pre-computed group scores from @DELPHI action tags and renders them as tables or bar charts. No statistical computation occurs inside the module.

Current Data Flow

Round 1 responses → Manual export to Excel → Researcher computes means → Paste scores into @DELPHI JSON → Module renders feedback

1.2 Proposed Architecture (v2.0)

The extension adds an auto-calculation engine as a server-side PHP layer. When calc: "auto" is set, the module extracts Round 1 responses, computes consensus metrics, and injects computed JSON into the existing JS renderers.

Proposed Data Flow

Round 1 data in REDCap → PHP Metrics Engine → Structured JSON → Existing JS renderers + badge

Key Design Principles

  • Backward compatibility: Missing or "manual" calc key preserves 100% current behaviour
  • REDCap EM framework compliance: All data access via REDCap::getData(), no direct SQL
  • No core modifications: Everything goes through hooks and the External Module API
  • Separation of concerns: Data extraction, metrics computation, caching, and rendering are distinct classes
  • Incremental delivery: 5 milestones, each independently testable and deployable

2. File-Level Analysis of Current Module

2.1 REDCapDelphi.php — The Entry Point (145 lines)

Hook Implementations

redcap_data_entry_form() and redcap_survey_page() both perform identical logic: check the current page is a data entry or survey page with a record ID, then include the ECharts library and delphi2.js, and call process().

process() Method — Core Logic

  1. Calls CheckForTags() to scan the current instrument's fields for any field whose misc (action tag) column contains @DELPHI
  2. If found, fetches the current record's data using REDCap::getData() in array format
  3. Iterates every field on the current page. For each field containing @DELPHI=:
    • Extracts the JSON substring after @DELPHI= (stripping any subsequent action tags)
    • Parses the JSON via json_decode()
    • Reads the display type (defaults to "single")
    • Resolves the preRound reference using a regex to extract [event][field] piping syntax
    • Looks up the participant's previous-round value from the fetched data
    • Reads the groups[] array (name, score, colour) verbatim from the JSON
  4. Builds a $config array and outputs it as a JavaScript global via setJsSettings()
Critical observation: The groups[].score values are read directly from the @DELPHI JSON and passed to the client unchanged. There is zero server-side computation. This is the exact insertion point for the auto-calculation engine.

preRound Resolution

The regex parses REDCap's piping syntax to extract the event name and field name. It then resolves the event ID via $Proj->getEventIdUsingUniqueEventName() and looks up the stored value. This mechanism is reused by the auto-calc engine to identify which Round 1 field to aggregate.

2.2 config.json — Module Manifest

Declares the module's identity and subscribed hooks. Currently minimal. For v2.0, this file must be extended with project-level settings fields for default stakeholder grouping field, default consensus rule, threshold, exclusion codes, and debug mode toggle.

2.3 scripts/delphi2.js — Client-Side Renderer

  • single / single-noprev: Injects a table above the radio button row showing stakeholder group names with coloured cells at the score position
  • single-matrix / single-matrix-noprev: Same as single but targets matrix-type question layouts
  • bar / matrix-bar: Creates ECharts bar chart instances showing distribution of responses per group

For auto-calc mode, delphi2.js needs no structural changes. The PHP engine will produce output in the same shape. The only addition is the consensus badge renderer.

3. REDCap Integration Points

3.1 Hook Lifecycle

The module uses two hooks that fire when a survey page or data entry form is being rendered. Both fire after the HTML form has been generated but before the page is sent to the browser, allowing the module to inject JavaScript.

3.2 Data Access

The module accesses REDCap data through: (a) the global $Proj object for project metadata, and (b) \REDCap::getData() for record-level data. For auto-calc, getData will be called additionally to fetch all records for the Round 1 event/field to compute aggregates.

3.3 Action Tag Parsing

Action tags are stored in each field's misc column. The module scans for @DELPHI= using string position search, extracts the JSON between = and the next @ character (or end of string), then parses it with json_decode().

3.4 JavaScript Injection

The module outputs <script src> tags for libraries and a <script>REDCapDelphi = {...};</script> block containing the serialised configuration. The JS file reads this global on DOMContentLoaded and manipulates the form DOM.

Step 2.3: Create lib/MetricsEngine.php

class MetricsEngine {
  public static function computeBasic(array $values): array {
    sort($values);
    $n    = count($values);
    $mean = array_sum($values) / $n;

    // Median
    $mid = (int)($n / 2);
    $median = ($n % 2 === 0)
      ? ($values[$mid-1] + $values[$mid]) / 2
      : $values[$mid];

    // Mode, IQR, SD, Distribution...

    return [
      'n' => $n, 'mean' => $mean,
      'median' => $median, 'mode' => $mode,
      'iqr' => $iqr, 'sd' => $sd,
      'distribution' => $distribution,
    ];
  }
}

Step 2.4: Modify REDCapDelphi.php process()

// Inside the field loop, after json_decode:
$calcMode = $params['calc'] ?? 'manual';

if ($calcMode === 'auto') {
  // Extract data → compute metrics → build groups array
  $extractor = new DataExtractor($project_id, $excludeCodes);
  $responses = $extractor->extractResponses($r1Event, $r1Field, $shField);

  // Build groups[] in same shape as manual mode

    $gm = MetricsEngine::computeBasic($vals);
    $groups['group'][] = ['name'=>$name, 'score'=>round($gm['mean']), ...];
  }
  $groups['consensus'] = $metrics; // extra data for badge
} else {
  // Existing manual mode (unchanged)
}

ConsensusRules.php pattern

class ConsensusRules {
  public static function evaluate(
    string $rule, array $metrics,
    int $scaleMin, int $scaleMax,
    float $threshold = 70.0
  ): array {
    return match($rule) {
      'iqr_sd'  => self::iqrSd($metrics),
      'range3'  => self::rangeConsensus($metrics, $scaleMin, $scaleMax, 3, $threshold, false),
      'range3_hi' => self::rangeConsensus($metrics, $scaleMin, $scaleMax, 3, $threshold, true),
      default   => ['status' => 'UNDETERMINED'],
    };
  }
}

Milestone 4: Stability Metrics + Status Badge

class StabilityMetrics {
  /** Coefficient of Variation for rating/ordinal data */
  public static function coefficientOfVariation(float $mean, float $sd): float {
    if ($mean == 0) return 100.0;
    $cv = ($sd / $mean) * 100;
    return round(100 - $cv, 2); // Group stability = 100% - CV
  }

  /** Percentage stability for categorical data */
  public static function percentageStability(
    array $distribution, int $consensusChoice, int $total
  ): float {
    $count = $distribution[$consensusChoice] ?? 0;
    return round(($count / $total) * 100, 2);
  }
}

Caching Strategy (MetricsCache.php)

  • Cache key: hash(project_id, event, field, stakeholderField, rule, threshold)
  • Storage: REDCap's built-in EM settings via setProjectSetting()/getProjectSetting()
  • Invalidation: Timestamp-based — recompute if project data has been modified since cache was written
  • Batch extraction: Single getData() call per Round 1 event, then compute per field in memory
