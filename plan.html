1. What the Module Does Today

The REDCap Delphi Module helps researchers run Delphi surveys inside REDCap. In a Delphi study, participants rate items in Round 1, then in Round 2 they see how the group responded before re-rating.

What the module handles well

  • Displaying group feedback (averages, distributions) directly inside the Round 2 survey
  • Showing the participant their own previous answer so they can compare
  • Multiple display formats: simple tables, matrix grids, bar charts
  • Separate feedback by stakeholder group (e.g. Patients, Clinicians, Nurses)

What the module does NOT do (the gap)

  • No automatic calculations — researchers must export data to Excel, manually compute averages and distributions, then paste those numbers back into REDCap field-by-field
  • No consensus measurement — no built-in way to determine if the group has reached agreement
  • No stability tracking — no way to measure how strong the agreement is
Why this matters: For a survey with 30 questions and 3 stakeholder groups, a researcher currently has to manually calculate and paste 90+ values. This is time-consuming, error-prone, and must be repeated if any Round 1 data changes.

2. What We Want to Build

We want to add an "auto mode" to the existing module. When enabled, the module will automatically pull Round 1 data from REDCap, compute all the statistics, and display them — no Excel needed.

How it works today vs. how it will work

Today (Manual Mode)Proposed (Auto Mode)
Step 1Researcher exports Round 1 data from REDCapModule reads Round 1 data automatically
Step 2Researcher opens Excel and calculates averages per groupModule computes mean, median, mode, IQR, etc. internally
Step 3Researcher types scores into each question's @DELPHI tagModule generates the display values automatically
Step 4Module displays the manually entered scoresModule displays the computed scores (same visuals)
Consensus?Not availableModule determines if consensus is reached (IN / OUT)
If data changes?Repeat steps 1–3 manuallyModule recalculates automatically
The key principle — nothing breaks: Any project already using the module with manually entered scores will continue to work exactly as before. Auto mode is only activated when a researcher explicitly adds calc: "auto" to their configuration.

3. What the Module Will Calculate

3a. Basic Statistics (for every question)

StatisticWhat It Tells YouExample
MeanThe average score across all respondents7.2 out of 9
MedianThe middle score when all responses are sorted7
ModeThe most frequently chosen score8
IQRThe spread of the middle 50% of responses (Q3 minus Q1)2
Standard DeviationHow spread out the responses are overall1.4
DistributionHow many people chose each score (1, 2, 3, … 9)Used for bar charts

These statistics are computed both overall and per stakeholder group.

3b. Consensus Rules

The module will support several standard Delphi consensus definitions. The researcher picks one rule for their study:

RuleHow It Decides Consensus
IQR + SD RuleConsensus if IQR ≤ 1 AND standard deviation ≤ 1.5
Range Consensus (2 ranges)Divide the scale in half. Consensus if 70%+ of responses fall in one half
Range Consensus (3 ranges)Divide the scale into Low/Medium/High. Consensus if 70%+ fall in one range
High-Only variantsSame ranges, but only the "High" range counts for consensus
70/15 RuleConsensus IN if 70%+ score High AND fewer than 15% score Low
MajorityConsensus if one choice gets more than 50% (absolute) or the most votes (simple)

Each rule produces a clear result: IN (consensus reached), OUT (consensus not reached), or UNDETERMINED.

3c. Stability Metrics

  • Coefficient of Variation (CV) — for rating questions. Lower CV = stronger agreement. Displayed as "Group Stability" percentage.
  • Percentage stability — for categorical questions. The percentage of people who chose the consensus answer.

4. What Participants Will See

The visual experience for survey participants stays almost identical to what they see today. The tables, bar charts, and matrix displays all look the same — the only difference is the numbers are computed automatically instead of typed in manually.

One addition: a small coloured badge next to each question's feedback showing the consensus status:

  • IN — Green — the group has reached consensus on this item
  • OUT — Red — the group has not reached consensus
  • UNDETERMINED — Amber — the result is borderline

This badge is optional — researchers can turn it on or off globally or per question.

5. How Researchers Configure It

5a. Project-Level Settings (set once)

In the External Modules settings page, the researcher sets defaults for the whole project:

SettingWhat It DoesDefault
Stakeholder grouping fieldWhich REDCap field identifies the respondent's groupNone (no grouping)
Consensus ruleWhich consensus definition to useIQR + SD
ThresholdThe percentage required for consensus70%
Exclusion codesValues to ignore (e.g. "99" for "I don't know")None
Show consensus badgeWhether participants see the IN/OUT badgeOff
Debug modeShows detailed computation info to admins onlyOff

5b. Per-Question Configuration

Each question can override the project defaults using the @DELPHI tag.

Manual mode (existing — unchanged): The researcher types in the scores by hand, exactly as today.

Auto mode (new): The researcher just specifies the Round 1 source field and sets calc to "auto". No scores needed — the module computes everything.

6. Technical Approach (Simplified)

The extension is built as new PHP classes inside the existing module. No changes to REDCap itself.

ComponentWhat It Does
DataExtractorPulls Round 1 responses from REDCap's database, filters out blanks and exclusion codes, groups by stakeholder
MetricsEngineComputes all the statistics: mean, median, mode, IQR, SD, distributions
ConsensusRulesApplies the selected consensus rule and returns IN / OUT / UNDETERMINED
StabilityMetricsCalculates group stability (CV or percentage)
MetricsCacheSaves computed results so the module doesn't recalculate on every page load
Consensus BadgeSmall front-end component that shows the coloured status badge

Data flow in auto mode

Participant opens Round 2 → Module detects calc: "auto" → DataExtractor fetches Round 1 → MetricsEngine computes stats → ConsensusRules checks agreement → Same tables/charts render

Safety measures

  • No individual data exposed: Only aggregated statistics reach the browser. No record IDs or individual responses are ever sent to the participant's device.
  • Caching: Computed results are cached so that a survey page with 30 questions doesn't compute 30 times.
  • Backward compatible: The manual mode code path is completely untouched.

7. Delivery Plan

The work is split into 5 milestones. Each one is independently useful and testable:

1 week
M1: Module Mapped

Verify current module works. Produce technical analysis document (done).

2–3 weeks
M2: Auto Summaries

Module computes mean/median/mode from Round 1 data and displays it. No more Excel for basic stats.

2 weeks
M3: Consensus Rules

Full set of consensus rules (IQR/SD, Range 2–5, High-only, 70/15, Majority). Each produces IN/OUT status.

1–2 weeks
M4: Stability + Badge

Group stability metrics. Consensus status badge in the UI. Caching. All project/question settings.

1 week
M5: Docs + Release

Updated README, demo REDCap project, test suite, version 2.0 release.

Total estimated effort: 7–10 weeks. Milestone 2 alone eliminates the Excel dependency — it could be released independently while the consensus rules are being built.

8. Discussion Points

  1. Which consensus rules are highest priority? Should we implement all of them or start with a subset (e.g. IQR/SD + Range 3)?
  2. Should the consensus badge be visible to participants by default, or only to researchers?
  3. What's the preferred default threshold — 70% as per COMET methodology, or study-specific?
  4. Is there a preference for how stakeholder groups are identified?
  5. Should we support more than one consensus rule per project?
  6. Are there any REDCap hosting constraints (PHP version, memory limits)?
  7. Would a batch configuration tool (auto-generate @DELPHI tags) be useful for v2.0?
  8. Who should be involved in testing the demo project before release?
